# Supabase — схема та міграції

Уся схема БД зберігається у файлах і відстежується в git. Кожна зміна структури =
новий файл міграції. Так ми завжди бачимо історію та можемо відтворити базу з нуля.

## Структура
```
supabase/
├── config.toml                 # конфіг Supabase CLI
├── migrations/
│   └── 20260610120000_init.sql # початкова схема (таблиці, RLS, ролі, тригери)
└── seed.sql                    # стартові дані (категорії, інгредієнти, налаштування)
```

## Встановлення CLI (одноразово)
```bash
brew install supabase/tap/supabase      # macOS
# або через npm без глобальної установки:  npx supabase <команда>
supabase --version
```

## Підключення до проєкту (після створення в дашборді)
```bash
supabase login                          # авторизація CLI
supabase link --project-ref <REF>       # REF з Dashboard → Project Settings
```

## Робота з міграціями
```bash
# Створити нову міграцію (порожній файл із таймстампом)
supabase migration new add_something

# Застосувати локально (запускає локальний Postgres у Docker)
supabase start
supabase db reset        # перестворює БД з усіх міграцій + seed.sql

# Застосувати на хмарний проєкт
supabase db push

# Підтягнути зміни, зроблені руками в дашборді, у файл міграції
supabase db pull
```

## Правила
- **Ніколи не редагуємо вже застосовану міграцію** — лише додаємо нову.
- Назва файлу: `<timestamp>_короткий_опис.sql` (CLI робить таймстамп сам).
- Перед push — рев'ю SQL у PR, як звичайний код.
- Секрети (Google OAuth, service-role) — у змінних оточення Vercel/Supabase, не в git.

## Що далі (наступні міграції)
- `create_order(...)` RPC: транзакційне створення `orders` + `order_items`
  з перерахунком суми та валідацією промокоду на сервері.
- Storage-бакет `products` (публічний) для фото товарів.
- Realtime/тригери для подій (за потреби).
