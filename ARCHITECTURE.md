# Суши-бар — Архитектура проекта

> Бюджетный интернет-магазин суши и роллов на React. Карточки товаров, корзина,
> оформление заказа без онлайн-оплаты, админ-панель для управления каталогом.
> Все сервисы — на постоянно бесплатных тарифах.

---

## 1. Цель и рамки (scope)

**Что делаем (MVP):**
- Витрина с категориями и карточками товаров.
- Фильтр/сортировка товаров **по ингредиентам** (ингредиенты — отдельная сущность).
- Корзина (клиентская) и оформление заказа без онлайн-оплаты (оплата при получении).
- Админ-панель: CRUD категорий, товаров, ингредиентов; привязка ингредиентов к товарам; просмотр заказов.

**Решения по бизнес-логике (зафиксированы):**
- **Язык интерфейса — только украинский.** Весь UI и контент (названия категорий, товаров, ингредиентов) — на украинском. i18n-слой не закладываем.
- **Получение заказа:** доставка **и** самовывоз (выбор при оформлении).
- **Минимальная сумма заказа:** нет.
- **Стоимость доставки:** конфигурируется в админке (`settings`), конкретные правила/тарифы пропишем позже. Архитектурно поле `delivery_cost` фиксируется в заказе.
- **Промокоды:** есть в MVP. Тип скидки — **процент** или **фиксированная сумма**.
- **Уведомления о новых заказах:** **Telegram-бот** (бесплатно, мгновенно).

**Что закладываем в архитектуру, но не реализуем сейчас:**
- Логика аккаунтов пользователей (пока единственная роль — `admin`).
- Онлайн-оплата (добавляется поверх готовой модели заказов позже).

**Принципы:**
- Только постоянно бесплатные тарифы (Supabase Free, Vercel Hobby).
- Минимум зависимостей, минимум внешних сервисов.

## 2. Технологический стек

| Слой | Решение | Тариф |
|------|---------|-------|
| Фреймворк | **Next.js (App Router)** + React + TypeScript | — |
| Стилизация | **CSS Modules** | — |
| БД | **Supabase Postgres** | Free (500 МБ) |
| Авторизация | **Supabase Auth** | Free |
| Хранилище картинок | **Supabase Storage** | Free (1 ГБ) |
| Уведомления о заказах | **Telegram Bot API** | Free |
| Хостинг | **Vercel** (Hobby) | Free |

**Почему так:**
- Next.js на App Router даёт SSR/SSG → хороший SEO для магазина, серверные actions для админки, нативную дружбу с Vercel и Supabase.
- Supabase закрывает БД + Auth + Storage одним бесплатным сервисом → меньше точек отказа и нулевая стоимость.
- CSS Modules — без рантайм-зависимостей, изоляция стилей, лёгкий бандл.

> ⚠️ **Особенность Supabase Free:** проект "засыпает" после ~7 дней полного бездействия.
> Для бюджетного проекта это приемлемо; при необходимости — лёгкий cron-пинг (Vercel Cron / GitHub Actions).

## 4. Модель данных

### ER-диаграмма (логическая)

```
categories ──< products >──< product_ingredients >── ingredients
                  │  │
                  │  └──< promos (акции/товар недели, спец-цена + баннер)
                  └──< order_items >── orders >── promo_codes (NULL)

reviews (модерация)   settings (key-value)   profiles ── auth.users
```

> **Категории vs акции.** Основная категория у товара — одна (`products.category_id`):
> роли / сети / супи / вок / соуси / буріто. «Акція тижня / товар недели» — это
> **накладывающаяся метка**, а не основная категория: реализуется через активную
> запись в `promos`. Поэтому товар может быть одновременно «супом» и «акцией недели»,
> «сетом» и «акцией» — категория не меняется, просто появляется промо-цена и баннер.

## 5. Безопасность (RLS — Row Level Security)

Включаем RLS на всех таблицах. Политики:

| Таблица | Чтение | Запись |
|---------|--------|--------|
| categories, products, ingredients, product_ingredients | публично (anon) | только `admin` |
| promos | публично — только активные (`is_active` + срок) | только `admin` |
| reviews | публично — только `approved` | **insert** — публично (создаётся `pending`), update/delete — `admin` |
| promo_codes | **нет прямого чтения** (проверка только через серверный action) | только `admin` |
| settings | публично (читаемые ключи, напр. стоимость доставки) | только `admin` |
| orders | `admin` + `editor` | **insert** — через серверный action; update — `admin`/`editor`; **delete — только `admin`** |
| order_items | `admin` + `editor` | insert вместе с заказом |
| profiles | сам пользователь / `admin` | только `admin` |
| allowed_staff | только `admin` | только `admin` |

> **Правило ролей (общее для контента — categories/products/ingredients/promos/promo_codes):**
> чтение публичное, `INSERT`/`UPDATE` — `admin` и `editor`, **`DELETE` — только `admin`**.
> Реализуется отдельными RLS-политиками на каждую операцию.

**Создание заказа** делаем через серверный action (Next.js) с полной валидацией на сервере:
1. Пересчёт `subtotal` по актуальным ценам товаров из БД (сумме с клиента не доверяем).
2. Валидация промокода: активность, срок (`valid_until`), лимит (`usage_limit`/`used_count`) → расчёт `discount`.
3. Расчёт `delivery_cost` по `settings` и типу получения (`delivery`/`pickup`).
4. Создание `orders` + `order_items` в транзакции (Postgres RPC), инкремент `used_count` промокода.
5. Отправка уведомления в Telegram (см. §8).

`service-role` ключ Supabase используется **только на сервере** (`lib/supabase/admin.ts`), никогда не попадает в браузер.

## 6. Корзина

- Клиентская, без записи в БД до оформления.
- Хранение: React Context + `localStorage` (переживает перезагрузку).
- При оформлении: серверный action принимает позиции → пересчитывает сумму по БД → создаёт `orders` + `order_items` в транзакции (RPC-функция Postgres).

## 7. Админ-панель

- Доступ: `/admin/*`, защищён проверкой сессии + роли в `layout.tsx`.
- **Аутентификация: Supabase Auth через Google OAuth.** Вход разрешён только если email
  пользователя есть в белом списке `allowed_staff`. Нет в списке → доступ запрещён.
- **Роли:** `admin` (главный — может всё, включая удаление) и `editor` (сотрудник —
  всё, кроме удаления). Роль берётся из `allowed_staff` / `profiles`.
  - Запрет удаления у `editor` — на двух уровнях: в UI (нет кнопок) и в БД (RLS:
    `DELETE` только при `role = 'admin'`).
- **Управление сотрудниками** (только `admin`): добавить email в `allowed_staff` с ролью,
  сменить роль, убрать из списка (мгновенный отзыв доступа).
- **Динамические категории:** добавленная категория сразу появляется в навигации сайта
  (шапка + бургер-меню) и как секция/фильтр меню. Навигация = закреплённые спец-пункты
  (Новинки, Акції) + категории из БД (`show_in_nav`, `sort_order`, `is_active`).
  Публичные страницы обновляются через revalidate.

> **Реализовано сейчас (каркас, без Supabase):** весь UI админки в стиле сайта —
> вход через Google (имитация), сайдбар, разделы, демо-переключатель роли для проверки
> ограничений `editor`, рабочий CRUD категорий (через клиентский стор, категории видны
> в шапке сайта) и управление сотрудниками. Данные/реальную авторизацию подключаем на этапе Supabase.

- Возможности:
  - **Категории:** CRUD, порядок, активность.
  - **Товары:** CRUD, загрузка картинки в Storage, привязка ингредиентов (множественный выбор), бейдж (ХІТ/НОВЕ), флаг «Хіт».
  - **Ингредиенты:** CRUD.
  - **Акции (`promos`):** CRUD, выбор товара, спец-цена, баннер, срок действия — это и слайдер «Акційні пропозиції», и «товар недели».
  - **Промокоды (`promo_codes`):** CRUD (процент/фикс, срок, лимит).
  - **Отзывы (`reviews`):** модерация — одобрить / отклонить.
  - **Настройки (`settings`):** стоимость доставки и пр.
  - **Заказы:** список, смена статуса.

## 8. Telegram-уведомления о заказах

Бесплатно и навсегда — через **Telegram Bot API**.

**Настройка (разово):**
1. Создаём бота через `@BotFather` → получаем `TELEGRAM_BOT_TOKEN`.
2. Узнаём `chat_id` админа (или общего чата) → `TELEGRAM_CHAT_ID`.

**Логика:**
- После успешного создания заказа серверный action делает один HTTP-запрос
  `sendMessage` с составом заказа (имя, телефон, тип получения, адрес, позиции, итог).
- Запрос идёт **только с сервера** (токен не попадает в браузер).
- Ошибка отправки в Telegram **не должна** ломать создание заказа — оборачиваем в try/catch и логируем; заказ уже сохранён в БД.

## 11. Дорожная карта (по этапам)

1. **Инициализация:** Next.js + TS + CSS Modules, украинская локаль, подключение Supabase-клиентов.
2. **Перенос верстки из прототипа** (§13): порт компонентов на CSS Modules с дизайн-токенами, добавление мобильной адаптивности.
3. **Схема БД:** миграции (все таблицы + RLS + RPC создания заказа), seed на основе данных прототипа.
4. **Витрина на реальных данных:** главная (hero, акции, хиты, меню), карточка/модалка товара из Supabase.
5. **Фильтр по ингредиентам** (нормализованные `ingredients`).
6. **Корзина + оформление заказа:** drawer → форма чекаута (доставка/самовывоз, адрес, промокод), расчёт итога на сервере, Telegram-уведомление. Отзывы → запись в БД.
7. **Админка:** auth → категории → ингредиенты → товары → акции → промокоды → отзывы → настройки → заказы.
8. **Деплой** на Vercel + Supabase, настройка Telegram-бота и Vercel Cron.

## 12. Принятые решения (вопросы закрыты)

| Вопрос | Решение |
|--------|---------|
| Язык интерфейса | Только **украинский**, без i18n-слоя |
| Получение заказа | **Доставка и самовывоз** (выбор при оформлении) |
| Минимальная сумма заказа | **Нет** |
| Стоимость доставки | Конфигурируется в админке (`settings`); тарифы пропишем позже, модель готова |
| Промокоды | Есть; тип скидки — **процент или фикс. сумма** |
| Уведомления о заказах | **Telegram-бот** |
| Картинки | **Supabase Storage**, публичный bucket |
| «Засыпание» Supabase Free | **Vercel Cron** раз в 3–5 дней |
| Роли админки | **admin** (всё) и **editor** (всё, кроме удаления) |
| Доступ сотрудников | **Google OAuth + белый список** (`allowed_staff`); добавление/удаление по email |
| Категории | **динамические** — добавил в админке → сразу в навигации сайта |

### Осталось уточнить только при необходимости (не блокирует разработку)
- Конкретные тарифы/зоны доставки и порог бесплатной доставки (значения в `settings`).
- Онлайн-оплата — отдельным этапом поверх готовой модели заказов.

Тёмная премиальная тема, шрифты Google Fonts (Cormorant Garamond + Jost).

**Что в прототипе уже готово:** витрина, фильтр по ингредиентам (по строковым меткам),

**Что дорабатываем:**

- **Отзывы** — форма не сохраняет; подключаем к `reviews`.
- **Фильтр** — строковые `ingredients` → нормализованная сущность `ingredients` + `product_ingredients`.

**Изображения** прототипа загружаем в Supabase Storage (bucket `products`) и используем
как стартовый контент/seed.
